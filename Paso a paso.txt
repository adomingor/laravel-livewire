        TENIA PROBLEMAS CON ALGUNAS FUNCIONES DEL wire:model (por ejemplo wire:model.blur o wire:model.change) consultando a Claudio...

En Livewire v3 (comportamiento antiguo):

Directiva	Comportamiento
wire:model.blur	Sincroniza con el servidor al perder el foco
wire:model.change	Sincroniza con el servidor al cambiar
En Livewire v4 (comportamiento nuevo - BREAKING CHANGE):

Directiva	Comportamiento
wire:model.blur	Solo actualiza el estado cliente (Alpine), NO envía al servidor
wire:model.change	Solo actualiza el estado cliente (Alpine), NO envía al servidor
wire:model.live.blur	Sincroniza con el servidor al perder el foco (= el viejo .blur)
wire:model.live.change	Sincroniza con el servidor al cambiar (= el viejo .change)
La lógica nueva en v4: los modificadores antes de .live controlan el timing del estado cliente (Alpine), los modificadores después de .live controlan cuándo se hace la petición al servidor.

    Cuando se crea un componente se puede pasarle parámetros sfc, mfc, class

    * SFC significa Single File Component (Componente de Archivo Único). En este formato, la lógica PHP (clase), la plantilla Blade (HTML) y, opcionalmente, los estilos CSS y scripts JavaScript, residen en un solo archivo con extensión .php o .blade.php, utilizando una sintaxis especial similar a los componentes de frameworks frontend como Vue.js.
    * MFC significa Multi-File Component (Componente de Múltiples Archivos). En este formato, la lógica PHP se encuentra en un archivo de clase (.php) separado de su respectiva plantilla Blade (.blade.php). Este era el enfoque convencional en versiones anteriores de Livewire (antes de v4).
    * Class se refiere específicamente al formato Class-based component (Componente basado en Clase). Es similar a MFC, pero se usa para referirse a la estructura tradicional donde el foco principal es la clase PHP que extiende de Livewire\Component y su vista asociada se encuentra en una ubicación separada, a menudo en resources/views/livewire

    El estándar de la industria para administrar roles y permisos en Laravel es el paquete spatie/laravel-permission.
        Características y Uso del Paquete Spatie:
            Instalación: Se instala fácilmente mediante composer require spatie/laravel-permission.
            Gestión: Permite crear roles (ej. 'admin', 'editor') y permisos (ej. 'editar artículos').
            Asignación: Se añaden permisos a roles ($role->givePermissionTo($permission)) y roles a usuarios ($user->assignRole('writer')).
            Verificación: Ofrece métodos sencillos en el modelo de usuario como $user->hasRole('admin') o $user->can('editar artículos').
            Middleware: Soporta middleware para proteger rutas directamente: Route::group(['middleware' => ['role:admin']], ...)
